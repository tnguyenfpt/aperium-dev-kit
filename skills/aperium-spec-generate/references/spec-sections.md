# Spec Sections Reference

## Overview

These section templates ensure specifications generated by the aperium-spec skill are compatible with the Speckit pipeline (`/speckit.specify` -> `/speckit.clarify` -> `/speckit.plan` -> `/speckit.tasks`).

## Required Sections

### 1. Feature Header

```markdown
# Feature Specification: {FEATURE_NAME}

**Feature Branch**: `{FEATURE_NUMBER}-{SHORT_NAME}`
**Created**: {DATE}
**Status**: Draft
```

### 2. User Scenarios & Testing (mandatory)

```markdown
## User Scenarios & Testing

### User Story 1 - {STORY_TITLE} (Priority: P1)

{STORY_DESCRIPTION}

**Acceptance Scenarios**:
1. **Given** {precondition}, **When** {action}, **Then** {outcome}
```

Aperium-specific guidance:
- Always consider MCP server interactions in scenarios
- Include scenarios for error handling and edge cases
- Reference the 6-layer defense model for security-sensitive features

### 3. Requirements (mandatory)

```markdown
## Requirements

### Functional Requirements
- **FR-001**: {REQUIREMENT_TEXT}

### Key Entities
- **{ENTITY_NAME}**: {DESCRIPTION}

### Assumptions
- {ASSUMPTION}
```

Aperium-specific guidance:
- Include requirements for PII handling if the feature processes user data
- Include RBAC requirements if the feature involves permissions
- Reference existing platform services (FastAPI, React, MCP) as context

### 4. Success Criteria (mandatory)

```markdown
## Success Criteria

### Measurable Outcomes
- **SC-001**: {MEASURABLE_CRITERION}
```

Aperium-specific guidance:
- Include performance criteria relevant to async FastAPI operations
- Include security criteria (coverage of 6-layer defense checks)
- Keep criteria technology-agnostic but domain-aware

### 5. Security Considerations (mandatory -- Aperium addition)

See `security-template.md` for the full template.

### 6. Edge Cases

```markdown
### Edge Cases
- What happens when {EDGE_CASE_SCENARIO}? ({EXPECTED_BEHAVIOR})
```

## Optional Sections

- **Non-Functional Requirements** -- performance, scalability, availability targets
- **Dependencies** -- external services, APIs, upstream/downstream systems
- **Out of Scope** -- explicit exclusions to prevent scope creep

### 7. Decisions (mandatory -- v2 addition)

```markdown
## Decisions

### Decision 1: {DECISION_TITLE}

- **Context**: {WHAT_PROMPTED_THE_DECISION}
- **Options considered**: {OPTIONS}
- **Chosen**: {CHOSEN_OPTION}
- **Rationale**: {WHY}
- **Trade-offs**: {WHAT_WAS_GIVEN_UP}
```

### 8. Review Checklist (mandatory -- v2 addition)

```markdown
## Review Checklist

- [ ] All acceptance criteria covered by tests
- [ ] Security considerations addressed (6-layer defense)
- [ ] PII handling verified (masking middleware applied)
- [ ] RBAC permissions configured correctly
- [ ] API contracts documented (OpenAPI/Pydantic models)
- [ ] Error handling covers edge cases
- [ ] Performance requirements validated
- [ ] No forbidden patterns introduced (see AGENTS.md)
```

## Compatibility Notes

- All sections use standard Markdown (no agent-specific syntax)
- Placeholder format: `{UPPER_SNAKE_CASE}`
- Speckit expects `##` headings for major sections, `###` for subsections
- The spec should be self-contained: a developer reading it should understand what to build without external context
